Indexing: R Starts at 1, Not 0!

Concept

In Python, the first element is x[0]. In R, the first element is x[1]. R starts counting from 1, like normal humans do.

Also, negative indexing works differently! In Python, x[-1] gives you the LAST element. In R, x[-1] gives you everything EXCEPT the first element. It REMOVES that position.

R Code

# Third element
ages[3]       # 19

# REMOVE the first element (negative index!)
ages[-1]      # 25, 19, 30, 22

# REMOVE first and second elements
ages[-c(1,2)]  # 19, 30, 22

sample() -- Picking Random Items

Business Story

Imagine you are running a raffle at a company event. You have 100 employee names in a hat. sample(employees, 5) picks 5 random winners (no repeats - you do not put the name back). But if you do sample(employees, 5, replace = TRUE), you put each name back after picking, so someone COULD win twice. This "with replacement" idea is CRITICAL for bootstrap later.

R Code
# Pick 3 random numbers from 1 to 10 (no repeats)
sample(1:10, 3)

# Pick 3 random numbers from 1 to 10 (repeats allowed)
sample(1:10, 3, replace = TRUE)

# Resample a whole dataset (this is bootstrap!)
data <- c(5, 8, 3, 12, 7)
boot_sample <- sample(data, length(data), replace = TRUE)
sample(1:10, 3) - Pick 3 random numbers from 1 through 10. Each number can only be picked once.
sample(1:10, 3, replace = TRUE) - Same but with replacement: after picking a number, "put it back." You might get repeats like 4, 4, 7.
sample(data, length(data), replace = TRUE) - Resample the data: pick as many items as the data has, with replacement. This is the core of bootstrap!

Level 2 - What Is a Hypothesis Test? (The Airport Story) +40 XP
Business Story

You land at San Diego airport. The sign says: "Average TSA wait time: 20 minutes." But you waited 35 minutes! Your friend waited 28. Another passenger waited 32. Is the airport LYING? Or did you just get unlucky?

You cannot just say "I waited long, so the sign is wrong." That is one person's experience. But you CAN collect data from 50 passengers and use a formal procedure to decide: is 20 minutes a reasonable claim, or is the real average higher?

That formal procedure is a hypothesis test.

Concept - Two Competing Claims

Every hypothesis test has two competing claims:

  • Null Hypothesis (H0): The boring default. Nothing special is happening. "The average wait IS 20 minutes."
  • Alternative Hypothesis (HA): The exciting claim. Something IS different. "The average wait is NOT 20 minutes" (or "is MORE than 20 minutes").

Think of it like a court trial: the defendant (H0) is "innocent until proven guilty." You need strong evidence to convict (reject H0).

Formula - Notation

Two-sided test (could be higher OR lower):

H0: mu = 20   vs   HA: mu != 20

One-sided test (only care about higher):

H0: mu = 20   vs   HA: mu > 20

One-sided test (only care about lower):

H0: mu = 20   vs   HA: mu < 20

mu (the Greek letter) = the TRUE population average (which we do not know)

Analogy - Two-sided vs One-sided

Two-sided: "Is the thermostat broken?" - Could be too hot OR too cold.

One-sided (greater): "Is the thermostat stuck too high?" - Only care about overheating.

One-sided tests are more powerful (easier to detect the effect) but you must decide the direction BEFORE seeing the data.

Level 3 - The p-value: What It Actually Means +50 XP
Business Story - The Coin Flip

Your friend says their coin is fair (50/50 heads/tails). You flip it 10 times and get 7 heads. Is the coin rigged? Well, even a fair coin could get 7 heads sometimes by luck. The p-value answers: "If the coin really IS fair, how likely is it to get a result THIS extreme (7 or more heads) just by chance?"

If that probability is tiny (like 1%), you think: "Wow, that would be really unlikely with a fair coin. The coin is probably NOT fair." If it is moderate (like 17%), you think: "Meh, 7 heads could easily happen by luck. I cannot claim the coin is rigged."

Concept - The p-value Definition

p-value = the probability of getting data this extreme (or MORE extreme), assuming H0 is true.

Small p-value = "This data would be very surprising if H0 were true" = evidence AGAINST H0.

Large p-value = "This data is not surprising under H0" = no strong evidence against H0.

Warning - What the p-value is NOT
  • NOT the probability that H0 is true
  • NOT the probability that HA is true
  • NOT the probability of making a Type I Error (that is alpha, which you choose)
  • NOT the probability that you will replicate the result

The p-value is ONLY about: "How extreme is my data, assuming H0 is true?"

Level 4 - The Decision Rule +30 XP
Concept - Alpha and the Decision

Before you even LOOK at your data, you pick a threshold called alpha (usually 0.05). Then:

  • If p < alpha: Reject H0. The data is surprising enough to doubt the null.
  • If p >= alpha: Fail to reject H0. The data is not surprising enough.

You must pick alpha BEFORE looking at the data. Otherwise you could cheat by picking alpha to match your desired result.

Analogy - The Jury

"Fail to reject H0" does NOT mean "H0 is true."

It is like a jury saying "not guilty." That does not mean the defendant is innocent - it means there was not ENOUGH evidence to convict.

You NEVER "accept H0" or "prove H0." You only ever "fail to reject" it.

Warning - Common Quiz Trap

If someone says "we accept the null hypothesis," that is WRONG language. The correct phrase is "we fail to reject the null hypothesis." This is almost certainly on the quiz.

Level 5 - Statistical Tests: The Big Picture (Before We Code Anything) +80 XP
Business Story — Why Tests Exist

You are a product analyst at Uber. The average ride takes 12 minutes. A new routing algorithm is deployed. After a week, you check: average ride time is now 11.4 minutes. Your VP asks: "Is that real improvement, or just random noise?"

You cannot just look at the number. 11.4 vs 12 could be genuine — or it could be chance variation in a sample. A statistical test is the formal tool that answers: "Could this difference have happened by luck alone?"

What Kinds of Tests Are There?

Concept — Types of Statistical Tests

Statistical tests are NOT one-size-fits-all. Before picking a test, ask two questions:

Question 1: How many groups am I comparing?

  • One-sample test: You have ONE group of data and you are comparing it to a KNOWN value. Example: "The airport claims average wait = 20 min. My sample of 50 passengers averaged 23.5 min. Is 23.5 different enough from 20?"
  • Two-sample test: You have TWO groups and you want to know if they differ. Example: "Group A saw the old checkout page, Group B saw the new one. Did Group B convert more?" This is the classic A/B test.

Question 2: Is my data roughly normal (bell-shaped), or is it skewed/weird?

  • Parametric test (e.g., t-test): Assumes data is roughly normal. Tests the MEAN.
  • Non-parametric test (e.g., sign test): Makes NO assumption about shape. Tests the MEDIAN.

Right: TWO-sample test. We have two groups (purple and blue). The question: "Is the gap between these two groups big enough to NOT be random noise?" This is exactly what an A/B test does.

Concept — Parametric vs Non-Parametric

Parametric = the test makes ASSUMPTIONS about the shape of your data (usually: "data is normal"). In exchange, it is more POWERFUL — it can detect smaller differences. The t-test is parametric.

Non-parametric = the test makes NO assumptions about shape. It works on ANY distribution (normal, skewed, bimodal, whatever). The trade-off: it is LESS powerful — it needs more data to detect the same difference. The sign test is non-parametric.

Think of it like this: a parametric test is a specialist doctor — great when the diagnosis fits, but wrong diagnosis = bad treatment. A non-parametric test is a general practitioner — works for everything, but might miss subtle issues.

Real Business Examples for Each Test Type

One-sample t-test: "Netflix claims average viewing session = 45 min. A sample of 200 users shows x̄ = 42 min. Is the claim wrong?"

One-sample sign test: "DoorDash tip distribution is heavily right-skewed ($0, $0, $0, $2, $3, $15, $50). Is the MEDIAN tip different from $3?"

Two-sample t-test: "A/B test at Spotify: Do users on the new homepage (Group B) stream more minutes per day than users on the old homepage (Group A)?"

Two-sample permutation test: "Amazon product ratings for sellers in California vs. Texas. Ratings are 1-5 stars (discrete, skewed). Do the distributions differ?"

Level 6 - The t-test: Visual Intuition, Formula, and R Code +80 XP
Business Story — The Airport Investigation

You collected wait times from 50 passengers at the airport. The airport claims average wait = 20 minutes. Your sample average is x̄ = 23.5 minutes. That is 3.5 minutes more than claimed. But is that enough to say the airport is lying?

Here is the problem: even if the TRUE average really IS 20 minutes, random sampling would give you a sample average that is NOT exactly 20. Maybe 19.2 one day, 21.1 another. The question is: is 23.5 far enough from 20 that it would be REALLY unlikely to happen by chance?

Visual: The t-test Idea

Reading This Chart — The Core Idea of a t-test

The blue bell curve shows: "If H₀ is true (the real mean IS 20), what sample means would we typically see?" Most sample means would land near 20 (the peak). Getting a sample mean of 16 or 24 would be rare.

The red line at 23.5 is our ACTUAL sample mean. It falls in the red rejection zone — the region that would be extremely unlikely if H₀ were true.

The t-test quantifies this: "How many standard errors away from 20 is our sample mean?" If it is far enough, we reject H₀.

The t-statistic Formula

Formula

t = (x̄ − μ₀) / (s / √n)

Let us decode every symbol:

  • x̄ (x-bar) = your sample average. In our example: 23.5
  • μ₀ (mu-zero) = the claimed/hypothesized value. In our example: 20
  • s = the sample standard deviation (how spread out your data is)
  • n = how many data points you collected. In our example: 50
  • s / √n = the standard error — this shrinks as n grows, making the test more sensitive

The t-statistic measures: "How many standard errors is my sample mean away from the claimed value?" A t-stat of 2 or more is usually far enough to reject.

Analogy — The GPS Analogy

Think of the standard error as the accuracy circle on your phone GPS. If your GPS says you are at position 23.5 with an accuracy circle of 1.5 (the standard error), and the airport claims you should be at 20.0 — well, 20.0 is WAY outside your accuracy circle. The claimed location does not match where you actually are. That is what a significant t-test tells you.

One-Sided vs Two-Sided Tests

Concept

Two-sided (default): H₀: μ = 20 vs Hᴀ: μ ≠ 20. "Is it different in EITHER direction?"

One-sided (greater): H₀: μ = 20 vs Hᴀ: μ > 20. "Is it LONGER than claimed?"

One-sided (less): H₀: μ = 20 vs Hᴀ: μ < 20. "Is it SHORTER than claimed?"

R Code — One-Sided Tests
# Default: two-sided (mu != 20)
t.test(tsa_waits, mu = 20)

# One-sided: is it GREATER than 20?
t.test(tsa_waits, mu = 20, alternative = "greater")

# One-sided: is it LESS than 20?
t.test(tsa_waits, mu = 20, alternative = "less")
When to Use Each Direction

Two-sided: You do not know which direction the difference might be. Most common in research. "We just want to know if it is different."

Greater: You specifically care about "is it BIGGER?" Example: "Do users on the new checkout page spend MORE time?" (Poker example from Lecture 5: testing if average winnings > 0.)

Less: You specifically care about "is it SMALLER?" Example: "Does the new caching layer reduce page load time?"

Level 7 - Type I/II Errors, Skewness, and When the t-test Breaks +100 XP
Business Story — The Fire Alarm and the Security Camera

Your office building has a fire alarm system. Two bad things can happen:

False alarm (Type I Error): The alarm screams "FIRE!" but there is no fire. Everyone evacuates, wastes 30 minutes, loses productivity. Annoying, but nobody gets hurt.

Missed fire (Type II Error): There IS a real fire, but the alarm stays silent. People stay inside. Dangerous.

In data science, replace "fire alarm" with "your statistical test" and "fire" with "a real effect in the data":

  • Type I Error: Your test says "the new feature works!" but it actually does NOT. You ship a useless feature. (You rejected H₀ when H₀ was true.)
  • Type II Error: Your test says "no difference" but the new feature DOES work. You kill a good feature. (You failed to reject H₀ when H₀ was false.)

Visual: The Four Outcomes

Concept — The Key Numbers

α (alpha) = the maximum Type I Error rate you are willing to tolerate. Usually 0.05 (5%). YOU pick this before looking at data.

β (beta) = the Type II Error rate. The probability you miss a real effect.

Power = 1 − β = the probability you correctly detect a real effect. Higher power = better at catching real signals.

Think of α as the "false alarm budget." Setting α = 0.05 means you accept that 5 out of 100 tests will falsely reject H₀ even when H₀ is true.

Industry Impact of Each Error Type

Type I Error at a tech company: You A/B test a new recommendation algorithm. The test says it is better (p = 0.03). You ship it. But it was actually NOT better — the 0.03 was a fluke. Result: wasted engineering effort, possible user experience degradation.

Type II Error at a pharma company: A new drug actually reduces blood pressure by 5 points. But your study was too small (n = 20), so the test says "no significant effect." The drug never gets approved. Patients miss out.

The trade-off: Making α smaller (e.g., 0.01 instead of 0.05) REDUCES Type I Errors but INCREASES Type II Errors (harder to detect real effects). It is a seesaw.

When Does the t-test BREAK? (This Is Exam Material!)

Concept — Two Conditions for the t-test

The t-test only works well when two conditions are met:

1. Normality: The data should come from a roughly normal (bell-shaped) distribution. This matters MOST when n is small (like 5 or 10). When n is large (100+), the Central Limit Theorem helps.

2. Independence: Each data point must not influence the others. Example of violation: measuring the SAME person twice, or stock prices on consecutive days.

Visual: Normal vs Skewed — Why This Matters

Reading This Chart

Left (Normal): The data is symmetric. The mean and median are the same. The t-test's math works perfectly here because the sampling distribution of x̄ will also be symmetric.

Right (Gamma/Skewed): The data has a long RIGHT tail. Notice the mean (yellow) is PULLED RIGHT by the extreme values, away from the median (cyan). This asymmetry is the problem.

Visual: HOW Type I Error Gets Inflated

This is the KEY insight that your professor tests on (Practice Quiz Q3a, Lab 2 Q1.1). Look at what happens when we take many small samples:

Warning — This Is Why Skewness Breaks the t-test

Panel A (Normal data, n=5): When we take 8,000 samples of size 5 from a normal distribution and compute x̄ each time, the sampling distribution is symmetric and centered at the true mean. The t-test assumes this shape. Everything works.

Panel B (Gamma/skewed data, n=5): Same thing, but from a skewed distribution. Now the sampling distribution of x̄ is ALSO skewed right! The t-test does NOT know this — it still assumes symmetry. So it puts the rejection regions in the wrong places. More sample means fall in the upper rejection region than expected, causing MORE false rejections (Type I Errors) than the promised 5%.

Panel C: As n increases, the Central Limit Theorem kicks in and the sampling distribution becomes more symmetric. The Type I Error rate gradually drops toward 0.05 — but even at n=30, it is still above 0.05 for this particular skewed distribution!

Concept — The Central Limit Theorem (CLT) Rescue

The CLT says: "As n gets larger, the sampling distribution of x̄ becomes more and more normal, REGARDLESS of the original distribution."

This is why the t-test eventually works even on skewed data — but ONLY with large enough n. With n=5 and skewed data, the CLT has NOT kicked in yet. The sampling distribution is still skewed, and the t-test's Type I Error rate is inflated above α.

Rule of thumb: For mildly skewed data, n ≥ 30 is often enough. For heavily skewed data (like the Gamma in your class), you might need n ≥ 50 or even 100.

What To Do When Data Is Skewed

Three Options When Normality Fails

Option 1: Use the sign test (non-parametric)

The sign test does NOT assume normality. It tests the MEDIAN instead of the mean. It works on ANY distribution. Trade-off: it is less powerful (needs more data to find the same effect).

Option 2: Get more data

If you can increase n, the CLT will eventually make the sampling distribution normal enough for the t-test to work. But this is not always practical.

Option 3: Use the bootstrap confidence interval

Resample your data with replacement, compute the mean each time, and use the percentiles of the bootstrap distribution. This is flexible but also imperfect — Lab 2 shows the bootstrap CI can ALSO have inflated Type I Error on skewed data with small n!

Real Example — Lab 2 Results (This Is On Your Exam)

In Lab 2, you simulated Type I Error rates for three tests on Gamma-distributed data:

  • t-test, n=5: Type I Error ≈ 0.08-0.09 (INFLATED above 0.05!)
  • Bootstrap CI, n=5: Type I Error ≈ 0.10+ (EVEN WORSE!)
  • Sign test, n=5: Type I Error ≈ 0.016 (below 0.05 — SAFE, but conservative due to discrete p-values)

The sign test was the ONLY test that controlled its Type I Error rate properly on this skewed data. This is the key takeaway from Lab 2.

Level 9 - Simulating Type I Error: The Master Pattern +70 XP
Business Story

Imagine you are testing 10,000 fire alarms. You KNOW there is no fire (H0 is TRUE). You want to count how many alarms go off anyway (false alarms = Type I Errors). If each alarm is set to alpha = 0.05, you would EXPECT about 500 false alarms out of 10,000 (5%).

In statistics, we do the same thing with SIMULATIONS. We generate fake data where we KNOW H0 is true, run a test, and check if it wrongly rejects. Repeat thousands of times and count the false alarm rate.

Concept - The Four-Step Pattern
  1. Generate fake data where H0 IS TRUE (e.g., rnorm with the claimed mean)
  2. Run a t-test on the fake data
  3. Check if we wrongly reject (is p < alpha?)
  4. Repeat 10,000 times and count how often we wrongly rejected
R Code - Full Type I Error Simulation
set.seed(42)
n <- 30                         # Sample size
n_sims <- 10000                 # Number of simulations
alpha <- 0.05                   # Significance level
mu0 <- 6                        # True mean (H0 IS true)
rejections <- 0                 # Counter for false alarms

for (i in 1:10000) {
  # Step 1: Generate data where H0 is TRUE
  fake_data <- rgamma(n, shape = 2, scale = 3)  # mean = 2*3 = 6 = mu0

  # Step 2: Run t-test
  p_val <- t.test(fake_data, mu = mu0)$p.value

  # Step 3: Check if we wrongly reject
  if (p_val < alpha) {
    rejections <- rejections + 1
  }
}

# Step 4: Calculate Type I Error rate
type1_rate <- rejections / n_sims
print(type1_rate)   # Should be near 0.05 if test works well

rejections <- 0 - Start counting false alarms at 0.

Level 10 - Bootstrap Confidence Intervals +60 XP
Business Story

You work at a coffee company. You have ONE bag of 50 beans. You want to estimate the average weight. You cannot order more bags (too expensive). So you scoop out 50 beans, weigh them, put them ALL BACK, shake, scoop 50 again, weigh. Repeat 10,000 times. Some beans get weighed multiple times - that is OK! The spread of your 10,000 averages gives you a confidence interval.

Concept - What is Bootstrapping?

Resampling with replacement = pick items from your data, but after each pick, put it back so it can be picked again. Your "bootstrap sample" has the same size as the original, but some values appear multiple times and some are missing.

Do this thousands of times, calculate the mean each time, and the middle 95% gives you a 95% CI.

R Code - Bootstrap CI
set.seed(42)
data <- c(12.1, 11.8, 13.2, 12.5, 11.9, 12.8, 13.1, 12.3)
n <- length(data)
B <- 10000                    # Number of bootstrap resamples
boot_means <- rep(0, B)       # Empty vector to store means

for (i in 1:B) {
  # Resample WITH replacement
  boot_sample <- sample(data, n, replace = TRUE)
  # Calculate mean of this resample
  boot_means[i] <- mean(boot_sample)
}

# 95% CI: the middle 95% of bootstrap means
ci <- quantile(boot_means, c(0.025, 0.975))
print(ci)
B <- 10000 - Do 10,000 bootstrap resamples.
boot_means <- rep(0, B) - Create vector of 10,000 zeros to fill.
sample(data, n, replace = TRUE) - Pick n values from data WITH replacement.
boot_means[i] <- mean(boot_sample) - Store the mean of this resample.
quantile(boot_means, c(0.025, 0.975)) - The 2.5th and 97.5th percentiles = 95% CI.
Using CI for Hypothesis Testing

If mu0 is OUTSIDE the 95% CI, reject H0. If mu0 is INSIDE the CI, fail to reject.

Example: CI = [11.9, 12.8]. If H0: mu = 12, since 12 is inside, fail to reject. If H0: mu = 15, since 15 is outside, reject.

Level 11 - The Sign Test: When Data Is Ugly +60 XP
Business Story

You are analyzing Yelp restaurant reviews. Scores are heavily skewed - lots of 5-star reviews, some 1-star, few in between. The mean is misleading. The MEDIAN is better. The t-test would be unreliable with this shape. You need a test that works for ANY distribution shape. That is the sign test.

Concept - The Sign Test Tests MEDIAN, Not Mean

The idea: if the true median is m0, then roughly half the data should be above m0 and half below. If way too many are above (or below), the median is probably not m0. No normality assumption needed!

R Code - Sign Test Function
sign_test <- function(data, m0) {
  # Remove values exactly equal to m0
  data <- data[data != m0]
  n <- length(data)

  # Count how many are above m0
  S <- sum(data > m0)

  # Two-sided p-value using binomial distribution
  p_val <- 2 * min(pbinom(S, n, 0.5), 1 - pbinom(S - 1, n, 0.5))
  return(p_val)
}
data[data != m0] - Remove values that equal the claimed median.
S <- sum(data > m0) - Count values above m0. data > m0 gives TRUE/FALSE, sum() counts TRUEs.
pbinom(S, n, 0.5) - Probability of S or fewer successes in n trials with p = 0.5.
2 * min(...) - Double the smaller tail probability for a two-sided test.
Warning - Discrete p-values

The sign test produces DISCRETE p-values. With n = 7, there are only a few possible p-values. You CANNOT always achieve exactly alpha = 0.05. This is tested in Lab 2!

sim_power_t <- function(n, mu0, delta, sd, n_sims = 10000, alpha = 0.05) {
  rejections <- 0
  true_mean <- mu0 + delta

  for (i in 1:n_sims) {
    data <- rnorm(n, mean = true_mean, sd = sd)
    p <- t.test(data, mu = mu0)$p.value
    if (p < alpha) {
      rejections <- rejections + 1
    }
  }
  return(rejections / n_sims)
}

Level 23 - Practice Quiz Q7-8: Big Data and Bootstrap +60 XP

Q7: The Big Data Trap

Q7 Walkthrough

A/B test with n = 2 million, version B wins by 0.001%, p = 0.0001. Switch to B?

No! Effect is real (p tiny) but practically meaningless. Always check:

  1. p-value: very small = effect is real
  2. CI: very narrow = high precision
  3. Cohen's d: very small = trivial effect
  4. Does this matter in practice? No.

Q8 Code
set.seed(123)
data <- c(23, 28, 19, 31, 25, 27, 22, 30, 26, 24)
mu0 <- 20
B <- 10000
n <- length(data)
boot_means <- rep(0, B)

for (i in 1:B) {
  boot_means[i] <- mean(sample(data, n, replace = TRUE))
}

ci <- quantile(boot_means, c(0.025, 0.975))
# Is mu0 = 20 inside CI? If outside: reject. If inside: fail to reject.
For 95% CI: c(0.025, 0.975). For 90% CI: c(0.05, 0.95).

Top 10 Traps to Avoid
  1. rnorm 3rd arg is SD, not variance
  2. 1:10 includes both endpoints (unlike Python)
  3. R indexing starts at 1, not 0
  4. Negative indexing REMOVES (not wrap-around like Python)
  5. "Fail to reject" not "accept H0"
  6. p-value is NOT P(H0 true)
  7. Small p + large n does not mean effect matters
  8. set.seed() must come BEFORE random functions
  9. Sign test tests MEDIAN, not mean
  10. : only does integer steps (use seq() for decimals)
